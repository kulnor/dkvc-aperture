## SystemNode

**Purpose:** xyflow custom node rendering a single map system tile (status stripe, security badge, tag, alias/name, Home marker, lock, statics/effect line) with inline edit affordances for alias and tag.
**File:** `src/components/map/SystemNode.tsx`

### Props
Receives xyflow `NodeProps` with `data: SystemNodeData` and `selected`.

`SystemNodeData` extends `MapSystemNode` with an optional `onAliasOrTagCommit(mapSystemId, field, next)` callback wired by `MapCanvas`, an optional `isHome` boolean (also set by `MapCanvas`, derived from the map's `homeMapSystemId`), and two optional load-time intel flags `inFactionWarfare` / `hasIncursion` (set by `MapCanvas` from the `intel` Record — `factionWar != null` / `incursion != null` — driving the bottom-right intel decorators). When the callback is present, the alias and tag chips render via `InlineTextEdit` (double-click to edit, Enter commits, Esc cancels, blur cancels, empty commits as null). When absent the chips fall back to plain spans (read-only path).

### Renders
A card surfaced with the `bg-map-node` token (a dedicated `--map-node` surface, set slightly lighter than `--card` in dark mode / a touch off pure white in light mode to ease eye strain against the canvas) with a thick left stripe coloured by the **security class** (`systemClassColor(data.security)`) and a resting 1px ring around the whole tile coloured by the **system status** (`systemStatusColor(data.status)`, composed into `box-shadow`), laid out as **two columns** (`flex items-stretch`). The **left column** is the security/class label stacked above the tag (separated from the right column by a `border-r`); both are `font-mono font-bold` coloured by `systemClassColor(data.security)` (the tag matches the class colour) — class at `text-lg`, tag at `text-sm` — and are the visual leads, sized up in their own column for legibility when zoomed out. The **right column** (`flex-1`) stacks the system name on top — `font-mono text-base text-foreground`, truncating to fill the width, with the optional pilot-presence badge, trade-hub pill, Home icon and lock icon trailing it — and a secondary line underneath. For wormhole systems that underneath line shows static target-class labels coloured via `systemClassColor` (e.g. "C3 C5") on the left, and — when the system has an anomaly effect — a small colour-coded effect square pinned to its right (`ml-auto`). For K-space systems it shows the region name.

### Behaviour & Interactions
- Drag handles on all four sides (top / right / bottom / left) are 12px dots tinted with the system's status colour (`handleBase`: 2px `--map-node` ring, `zIndex: -1`) that **fade in only while the tile is hovered** (`handleClass`: `opacity-0 … group-hover:opacity-[0.85]`, driven by `group` on the tile root); xyflow `nodesConnectable` / `nodesDraggable` are controlled by `MapCanvas`. Per-side transforms (`handleStyles`) push the bulk of each dot outside its edge while the inner sliver tucks under the tile, and `zIndex: -1` paints them behind the node so they never cover tile content. The left dot carries an extra `-4px` in its transform to clear the thick 4px class-coloured stripe drawn on that edge (otherwise it reads as misaligned). All four are declared as `type="source"` and the canvas runs in `ConnectionMode.Loose` so any side can act as either end of a new connection. `ConnectionEdge` picks which two sides to render against at draw time based on relative node centres, so the stored handle pair is incidental.
- Each handle carries a **unique `id`** (`top` / `right` / `bottom` / `left`). This is load-bearing for the connection-drag ghost line: xyflow's `getHandle` falls back to `handles[0]` (the first-declared handle = Top) whenever the handle id is null, which would pin both the drag **origin** (`fromHandle`) and the **snap target** (`toHandle`) to the top handle regardless of which side the user actually grabbed or is hovering. With ids, the ghost connection starts from the dragged side and snaps to the hovered/closest handle on the destination. The created connection itself ignores `sourceHandle`/`targetHandle` (`onConnect` uses only node ids; edges aren't handle-bound), so the ids affect only the in-progress preview.
- Selection is reflected by a prominent halo in the system's status colour — an offset solid outline (2px, 3px offset) plus a soft outer glow (`box-shadow` with alpha'd status hex) and a slight `scale(1.04)` lift, transitioned over 150ms. The larger spread, glow and outline distinguish it from the 1px resting status ring (same colour). Selection state is owned by `MapCanvas`. The card uses `cursor-pointer` so the entire tile reads as clickable — any click bubbles through xyflow's node wrapper to fire selection.
- **Home marker** (`isHome`): the map's designated Home system renders a gold/amber accent ring around the tile plus a `Home` (house) icon in the head row (colour from `homeAccentColor`). The ring is composed into the same `box-shadow` as the resting status ring and the selection halo (concentric, increasing spread), so a selected Home tile shows all three; an unselected, non-Home tile shows just the status-coloured resting ring. Home identity is a map-level property (`ap_map.home_map_system_id`) propagated at load time, not over realtime, so the marker reflects load-time state.
- Wormhole detection: has statics, or name matches `J######`.
- **Display name:** the alias text (and its placeholder when no alias is set) uses `systemDisplayName(data.systemId, data.name)`, which substitutes the short community name for the five Drifter systems (e.g. "Barbican" for "Liberated Barbican"); every other system shows its stored `name`. Display only — the stored name is unchanged.
- Inline editors carry `nodrag nopan` (set inside `InlineTextEdit`) so editing doesn't trigger pan / drag.
- **Trade-hub proximity badge:** when `data.tradeHub` is set (an HS system within a hub's high-sec-only proximity radius, precomputed at SDE ingest — see `universe_system.nearest_trade_hub_id/jumps`), the head row renders an emerald `IndicatorPill` reading `{jumps}j` (e.g. "3j") with a tooltip "N jump(s) to <hub>" (e.g. "1 jump to Jita"). Replaces the old manual workaround of renaming a system's alias to embed the distance. Sits between the presence badge and the Home icon. Null → nothing rendered.
- **Pilot-presence badge** (`PresenceBadge`, internal): reads `usePresenceForSystem(data.systemId)` (see `MapPresenceContext`). Renders nothing when the count is zero. Otherwise renders a small rounded-pill button (Users icon + count) wrapped in Base UI `PreviewCard`. Hover or focus opens a popup listing every online tracked pilot in the system as `name — shipName` with the ship *type* name on a secondary muted line beneath. The type line is omitted when the pilot never renamed the hull (ESI defaults `ship_name` to the type name, so `shipName === shipTypeName`); falls back to `shipTypeName` then `—` when no custom name is known. The trigger and popup both carry `nodrag nopan` so the hover interaction never starts a canvas pan/drag.
- **Signature indicators** (`SignatureIndicators`, internal): reads `useSignatureIndicator(data.id, isWormhole)` (see `MapSignatureIndicatorContext`). Floats just off the top-right corner (`absolute -top-2 -right-2`). The wrapper is `pointer-events-none nodrag nopan` so the gaps never swallow a node click. Renders a **clock** pill (amber) when the system is stale — its newest sig is older than the viewer's threshold, or a *wormhole* system has no sigs — with a compact age (`3h`, none for an empty system); and a **signal** pill (sky) with a count when the system has unscanned sigs (no group, or a wormhole missing type / "leads to"). Both honour the viewer's per-account toggles. Renders nothing when neither applies. The clock re-evaluates on a 60s tick from the context.
- Each pill is an `IndicatorPill` (internal) — a Base UI `Tooltip` whose trigger re-enables `pointer-events-auto` (overriding the wrapper) so a hover/focus tooltip can open above it. The clock tooltip reads "Signatures last updated 3h ago" (or "No signatures scanned" for an empty system); the signal tooltip reads "N unscanned signature(s)". Both popups carry `nodrag nopan`.
- **Intel indicators** (`IntelIndicators`, internal): mirrors `SignatureIndicators` (same `IndicatorPill` look, off-border float, `pointer-events-none nodrag nopan` wrapper) but pinned to the **bottom-right** corner (`absolute -right-2 -bottom-2`) so the two indicator sets never collide. Renders a **swords** pill (orange) when `data.inFactionWarfare` is set (tooltip "Faction Warfare system") and a **radiation** pill (red) when `data.hasIncursion` is set (tooltip "Active incursion"). Each is icon-only (no count/age text). Both flags are fed from load-time intel via `MapCanvas`, so they reflect map-load state (no realtime) like the Home marker. Renders nothing when neither applies.
- **System-effect indicator** (`EffectIndicator`, internal): for a W-space system whose `data.effect` is set (one of the six `SystemEffectKey`s), a small colour-coded square (`systemEffectColor`) sits at the bottom-right of the footer row (`ml-auto`, same row as the statics). Wrapped in a Base UI `PreviewCard`; hover or focus opens a panel headed by the effect name (`systemEffectName`) listing every bonus + value resolved to this system's class (`systemEffectBonuses`, fed the class number parsed from `data.security` via `classIdFromSecurity`). Trigger and popup carry `nodrag nopan`. `data.effect` is populated by the SDE ingest from `SYSTEM_EFFECT_BY_ID` (CCP's SDE omits it), so it reflects load-time static data.
- **System-kind icons** (`SystemKindIcon`, internal): two head-row badges that flag a special wormhole-system kind which isn't obvious from the J-sig. A **shattered** system (`isShatteredSystem(data.systemId)`, `@/lib/eve/shatteredSystems`) shows a rose `CircleDashed`; a **Drifter** system (`isDrifterSystem(data.systemId)`, `@/lib/eve/drifterSystems`) shows a violet `Atom`. Both sit just before the Home icon. Each is a bare lucide icon wrapped in a Base UI `Tooltip` (trigger is a `span` so clicks still bubble to node selection; `nodrag nopan` keeps the hover from panning) reading "Shattered system" / "Drifter wormhole". Detection is by solar-system id, so the two sets are mutually exclusive — the Drifter systems are excluded from the shattered set, and Thera reads as shattered.
- All other field edits (status, intel notes, locked, rally, signatures, remove) live in `InspectorModule`.
- **Underglow:** the tile is `relative`; two glow layers render behind the card. (1) **Transient** — when `useUnderglowForSystem(data.id)` returns an active glow, a `SystemUnderglow` is rendered keyed by the store's `token` so a rapid re-trigger restarts the animation (killmail / ping, via `MapUnderglowBridge`). (2) **Persistent rally** — when `data.rallyAt` is set, a `SystemUnderglow` with the `RALLY_UNDERGLOW` preset (amber, slow pulse) is rendered directly from map state, so it lives exactly as long as the rally is designated and a coinciding transient glow can't clear it. The two layers can coexist on one node.
